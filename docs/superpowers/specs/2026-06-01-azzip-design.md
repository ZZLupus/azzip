# azzip · 现代解压/压缩管理器 — 设计文档

**日期**:2026-06-01
**状态**:设计已确认,待写实现计划
**一句话**:一个无广告、UI 现代、轻量小巧的 Windows 解压/压缩管理器,用 Tauri 构建,架构为跨平台预留。

---

## 1. 背景与动机

市面解压软件普遍存在两个痛点:充斥广告、UI 陈旧丑陋。本项目从零构建一个干净、漂亮、不打扰用户的替代品。

**核心取向(贯穿所有决策):**
- **无广告、无捆绑、不偷抢文件关联** —— 与臃肿软件划清界限
- **小巧轻量** —— 安装包小、内存占用低
- **UI 现代漂亮** —— 这是最高优先级的差异化点
- **尊重用户** —— 默认行为克制,关键动作征求同意

---

## 2. 范围

### V1 功能范围
- **解压**(核心,最高优先级):双击/右键解压压缩包
- **创建压缩包**:选中文件 → 打包为 zip/7z 等
- **包内预览**:不解压即可浏览包内文件、预览内容、选择性提取单个文件
- **密码压缩包**:加密包的解压与创建

### 格式范围
- **读写**:ZIP、7z、TAR、GZ(及常见组合如 tar.gz)等开放格式
- **仅解压**:RAR(RAR 创建为专有授权,不做)

### 平台范围
- **V1 目标**:Windows(开发者当前环境 Windows 11)
- **架构要求**:跨平台预留 —— 系统集成代码模块化隔离,未来扩展 macOS/Linux 时只需新增平台实现,不推翻 UI 与核心逻辑

### 产品性格
- **两者都要**:既有轻量的右键/双击快捷路径,也有一个漂亮的主窗口管理器

### 明确不做(YAGNI)
- RAR 创建、大量冷门古董格式(cab/iso/lzh 等)
- macOS/Linux 的 V1 实现(仅预留架构)
- 云存储集成、文件同步等超范围功能

---

## 3. 技术栈决策

### 框架:Tauri(已确认)
Rust 后端 + 系统 WebView 渲染前端(HTML/CSS/JS + TS,可配 React/Vue/Svelte)。

**选型理由:**
- 安装包极小(~3-10MB)、内存占用低 —— 契合"小巧轻量"取向
- UI 用 Web 技术 —— 充分发挥开发者前端背景优势,最易做漂亮
- 跨平台是一等公民 —— 契合跨平台预留目标
- Rust 后端性能好、安全

**对比与排除:**
- **Electron**:全 JS 上手快、生态成熟,但安装包大(80-150MB 内置 Chromium)、内存占用高,与"轻量"取向冲突 —— 排除
- **Flutter/.NET/Qt 等原生路线**:放弃前端优势,UI 调优更难 —— 排除

### 解压引擎:纯 Rust 库(已确认)

| 格式 | crate | 能力 |
|------|-------|------|
| ZIP | `zip` | 读+写,密码(AES/ZipCrypto) |
| 7z | `sevenz-rust2` | 读+写,密码 |
| TAR | `tar` | 读+写 |
| GZ/BZ2/XZ | `flate2` / `xz2` 等 | 读+写,常与 tar 组合 |
| RAR | `unrar`(绑定官方 unrar 源码) | 仅解压 |

> 注:具体 crate 名称在实现阶段以最新可用、维护活跃的版本为准,此处为选型方向。

**选型理由:**
- 纯 Rust 静态编译进二进制,**安装包小、零外部依赖、无 DLL**,契合"小巧轻量"
- 跨平台编译干净
- 覆盖全部目标格式,无短板
- Windows 上无需 C 编译工具链

**对比与排除:**
- **libarchive(C 库)**:一套 API 吃遍所有格式、覆盖最全,但引入 C 依赖、Windows 打包复杂、创建/密码不如纯 Rust 顺手,且违背"无外部依赖"取向 —— 排除。唯一会改推的场景是未来需大量冷门格式,但不在范围内。
- 多库 API 不统一的唯一缺点由 `ArchiveHandler` trait 抽象抹平。

## 4. 整体架构

Tauri 应用分两层,职责清晰隔离:

```
┌─────────────────────────────────────────────┐
│  前端 (WebView)                                │
│  React/Vue/Svelte + TS                        │
│  · 主窗口管理器 UI(浏览/拖拽/预览)            │
│  · 进度、错误、密码、冲突等交互弹窗            │
│  · 通过 Tauri "命令" 调用后端,监听"事件"      │
└───────────────┬───────────────────────────────┘
                │  invoke(命令)  /  emit(事件)
┌───────────────┴───────────────────────────────┐
│  Rust 后端  —  薄胶水 + 引擎                    │
│                                                │
│  ┌─────────────┐  ┌──────────────┐            │
│  │ commands.rs │  │  archive/     │  引擎抽象   │
│  │ (暴露给前端) │→ │  - trait 接口 │            │
│  └─────────────┘  │  - zip.rs     │            │
│                   │  - sevenz.rs  │            │
│  ┌─────────────┐  │  - tar/gz.rs  │            │
│  │ platform/   │  │  - rar.rs(解压)│           │
│  │ (系统集成)   │  └──────────────┘            │
│  │ - 右键菜单   │                              │
│  │ - 文件关联   │   ← 跨平台预留:每平台一份实现 │
│  │ - 拖拽/双击  │                              │
│  └─────────────┘                              │
└───────────────────────────────────────────────┘
```

### 三个关键设计决定

**1. 引擎抽象成 trait(`ArchiveHandler`)**
每种格式实现统一接口:`list / extract / create / extract_one`。前端永不关心具体格式 —— 它只发"解压这个文件",后端按扩展名/魔数路由到对应实现。加新格式 = 加一个文件,不动其余代码。

**2. 系统集成单独成 `platform/` 模块**
右键菜单、文件关联、双击行为等是唯一平台相关代码。V1 只写 Windows 实现,但接口定好,未来加 macOS/Linux 仅填新文件,不推翻 UI 与核心。这是跨平台预留的落点。

**3. 长任务走异步 + 事件**
解压/压缩大文件耗时,不能阻塞 UI。后端在异步任务中执行,通过事件持续 `emit` 进度;前端订阅显示进度条。密码错误、文件冲突等也走此通道回传。

## 5. UI 与界面设计

### 视觉风格(已通过 mockup 确认)
- **方向**:C 渐变个性 + B 毛玻璃质感 + 深色基调
- **主色**:紫粉渐变(`#a78bfa → #f0abfc`)—— 可配置,后续可换色系
- **背景**:深紫径向渐变(`#241b4d → #0e0c1f`)
- **质感**:半透明毛玻璃面板(`backdrop-blur`)、圆角卡片化列表行、光晕装饰
- **主题**:V1 深色为主,**后续支持浅色主题**(设计时变量化配色,预留切换)

### 主窗口
标题栏(含"解压全部""新建"按钮)+ 面包屑导航 + 包内搜索框 + 文件列表(名称/大小/类型列,选中态紫色高亮)+ 底部状态栏(就绪状态、项目数、压缩率)。

### 关键交互流程(已通过 mockup 确认)
1. **解压进度浮窗**:轻量,带进度条(发光)、速度、剩余时间、当前文件、取消
2. **密码输入弹窗**:加密包自动弹出,锁图标,密码错误内联提示+抖动重试
3. **拖拽压缩**:拖文件入窗 → 虚线高亮"松手即压缩",可选目标格式(ZIP/7z/TAR.GZ)
4. **包内预览**:双击文件不解压即看内容,支持"仅解压此项"

视觉语言全程统一(快捷路径与主窗口共用一套设计语言)。

## 6. Windows 系统集成(已确认)

| 集成点 | 行为 |
|--------|------|
| **双击压缩包** | 可设置;**默认打开主窗口**浏览,设置中可改为"直接解压到同名文件夹" |
| **右键菜单** | 丰富快捷项,**全部聚合进 `azzip →` 二级子菜单**,不污染右键主菜单 |
| **文件关联** | **首次启动询问**用户是否设为默认打开程序,不静默抢占 |

### 右键子菜单内容
- 右键**压缩包**时 `azzip →`:解压到此处 / 解压到同名文件夹 / 用主窗口打开
- 右键**文件或多选**时 `azzip →`:添加到压缩包… / 压缩为 "xxx.zip" / 压缩为 "xxx.7z"

### 实现成本备注
Win11 现代右键二级子菜单需 `IExplorerCommand` + 打包成 MSIX / 稀疏包(`appxmanifest`),比老式注册表写法复杂,但这是 Win11 唯一正规途径。实现阶段需为此预留分发打包方案。

## 7. 错误处理与边界情况(已确认)

**核心原则:永远给人话,不给错误码;失败要么干净回滚,要么明确告知部分结果。**

| 场景 | 处理策略 |
|------|----------|
| 密码错误 | 弹窗内联提示"密码不正确",输入框抖动,可重试,不关闭弹窗 |
| 磁盘空间不足 | 解压前预估所需空间,不足则提前拦截,不留半截文件 |
| 文件名冲突 | 弹窗询问:覆盖/跳过/两者保留(重命名),可"应用到全部" |
| 压缩包损坏/不完整 | 明确报错,能解多少解多少(可选),不静默失败 |
| 不支持的格式/压缩方法 | 友好提示具体原因,而非神秘错误码 |
| 路径过长/非法字符 | 自动处理长路径(`\\?\` 前缀),非法字符提示 |
| 中途取消 | 清理已解压的部分文件,回到干净状态 |
| 权限不足 | 提示并引导选其他目录,或请求提权 |

## 8. 测试策略(已确认)

Tauri 前后端分离,分两层测试。**原则:后端引擎层追求高覆盖(命脉),前端与 E2E 覆盖关键路径即可,不追求虚高数字。**

**Rust 后端(核心,重点测)**
- 引擎层单元测试:每种格式准备测试样本包,验证 `list/extract/create/extract_one`
- 边界用例测试:加密包(对/错密码)、损坏包、空包、超长路径、文件名冲突、中途取消回滚 —— 第 7 节每个边界都变成一条测试
- trait 契约测试:同一组断言跑遍所有引擎实现,保证抽象一致

**前端(交互层,轻量测)**
- 关键组件状态逻辑单元测试(文件列表、进度条、密码弹窗)
- 优先保证"命令调用 + 事件订阅"胶水逻辑正确

**端到端(少量,黄金路径)**
- Tauri WebDriver 驱动:"打开包→浏览→解压""拖拽→压缩""双击预览"

**方法论**:遵循 superpowers test-driven-development,先写测试再写实现。引擎层尤其适合 TDD。

## 9. 难度评估小结

| 部分 | 难度 | 说明 |
|------|------|------|
| 解压/压缩算法本身 | 低 | 调用成熟 Rust 库,非自研 |
| 漂亮现代 UI | 中高 | 核心差异化点,需在设计系统/动效/细节上花心思 |
| 无广告干净体验 | 低 | 产品决定,非技术难题 |
| Windows 系统集成 | 中 | `IExplorerCommand` + MSIX 打包稍琐碎但成熟 |
| 跨平台预留 | 低(架构层) | 模块化隔离,V1 不实现其他平台 |

**总体可行性:高。** 核心风险与精力应集中在 UI 打磨与 Win11 系统集成打包两处。
